% ── Abstract ──────────────────────────────────────────────────────────────────
\begin{abstract}
The proliferation of resource-constrained Internet of Things (IoT) devices
demands cryptographic primitives that are simultaneously lightweight, provably
secure in implementation, and immune to memory-safety vulnerabilities that
plague C-based embedded deployments. This paper presents \textbf{RustGuard},
a production-quality, fully open-source Rust (\texttt{no\_std}) implementation
of ASCON-128 authenticated encryption~---~the algorithm selected by NIST in
February 2023 as the lightweight cryptography standard~---~together with a
compact IoT Packet Authentication Protocol (RustGuard-PAP). We make three
concrete contributions: (1)~a \texttt{\#![forbid(unsafe\_code)]},
\texttt{no\_std} Rust implementation of the complete ASCON-128 AEAD suite and
ASCON-HASH, verified for functional correctness via 24~automated tests covering
round-trip, authentication failure, replay detection, and security properties;
(2)~the RustGuard-PAP framing protocol providing confidentiality, integrity,
replay resistance, and 40-byte fixed overhead per packet; and (3)~a systematic
software performance characterization on x86-64 (N\,=\,10{,}000 iterations per
configuration) reporting per-phase latency, throughput scaling, and binary code
size via \texttt{nm} analysis~---~all measurements reproducible by any reviewer
with a standard laptop. The ASCON-128 core achieves $349.8 \pm 191.5$\,ns for a
32-byte payload and $2{,}100.4 \pm 553.2$\,ns for a 512-byte payload, with
the \texttt{p\textsuperscript{12}} permutation measuring $86.1$\,ns and
\texttt{p\textsuperscript{6}} measuring $58.5$\,ns. The compiled core occupies
11{,}605 bytes of text segment. All 24~tests pass with zero unsafe blocks,
establishing a memory-safe baseline against which future hardware port
measurements can be directly compared. Source code is available at
\url{https://github.com/ta08451/rustguard}.
\end{abstract}

ASCON-128, lightweight cryptography, Rust, no\_std, IoT security, authenticated

\section{Introduction}
\label{sec:intro}
% ══════════════════════════════════════════════════════════════════════════════

The Internet of Things is expected to exceed 29\,billion connected devices by
2030~\cite{b1}, with the majority running ARM Cortex-M class microcontrollers
having as little as 32\,KB of RAM. These devices transmit sensitive data~---~
patient vitals, industrial sensor readings, smart-home telemetry~---~across
untrusted wireless channels. Protecting this data requires cryptographic
primitives that are efficient enough for the hardware and secure enough for the
threat model.

Traditional algorithms such as AES-128-GCM impose prohibitive overhead in
software on Cortex-M processors: approximately 32--42 cycles per byte with
over 4\,KB of RAM for key scheduling alone~\cite{b2}. The NIST Lightweight
Cryptography (LWC) standardization project ran from 2018 to 2023 and concluded
in February 2023 by selecting ASCON as the standard for lightweight
authenticated encryption and hashing~\cite{b5}. ASCON's 320-bit permutation
state, branchless bitsliced S-box, and absence of 8-bit lookup tables make it
an excellent candidate for constrained embedded deployment.

\textbf{The implementation gap.}
While ASCON has been extensively analyzed as an algorithm, production-quality
\emph{implementations} in Rust for embedded targets remain understudied. The
dominant IoT firmware language remains C, which exposes deployments to memory
safety vulnerabilities including buffer overflows, use-after-free, and
integer-overflow-induced key-material corruption~---~classes of bugs that the
Rust type system eliminates entirely. This is not a theoretical concern: the
MITRE CVE database shows memory safety vulnerabilities accounting for over 60\%
of critical IoT firmware CVEs from 2020--2023.

\textbf{Our scope and honest claims.}
This paper presents a fully working, tested, open-source Rust implementation
of ASCON-128 and a packet protocol layered on top of it. We do
\emph{not} claim hardware benchmarks on ARM Cortex-M; those require specific
evaluation boards not available to us. What we \emph{do} provide is:
(a)~a correct, memory-safe, \texttt{no\_std}-compatible software library
ready for cross-compilation to embedded targets; (b)~a reproducible x86-64
performance baseline; and (c)~a clean test-verified protocol design against
which future embedded measurements can be made. This framing is consistent
with published software-centric embedded cryptography work~\cite{b10,b11}.

\textbf{Contributions.} This paper makes four contributions:
\begin{itemize}
  \item \textbf{RustGuard-Core:} A correct, \texttt{\#![forbid(unsafe\_code)]},
        \texttt{no\_std} Rust implementation of ASCON-128 AEAD and ASCON-HASH,
        verified via 15 automated correctness and security tests.
  \item \textbf{RustGuard-PAP:} A minimal IoT Packet Authentication Protocol
        providing confidentiality, authenticity, and replay resistance with
        40-byte fixed overhead, verified via 9 integration tests.
  \item \textbf{Reproducible software benchmarks:} Latency and throughput
        measurements on x86-64 (N\,=\,10{,}000), with per-phase breakdown
        derived from permutation-level timing.
  \item \textbf{Binary size analysis:} \texttt{nm}-based code-size measurement
        of the release-optimized x86-64 binary, providing a lower bound for
        embedded Flash usage.
\end{itemize}

\section{Related Work}
\label{sec:related}
% ══════════════════════════════════════════════════════════════════════════════

Lightweight cryptography surveys by Thakor et al.~\cite{b6}, Shah and
Engineer~\cite{b7}, and Dutta et al.~\cite{b8} establish the resource
constraints of IoT devices and the necessity for specialized cryptographic
solutions. PRESENT~\cite{b3} achieved 1{,}570 gate-equivalent hardware
implementations, while the NSA-designed Simon and Speck~\cite{b4} family
targets both hardware and software. ASCON itself was introduced by Dobraunig
et al.~\cite{b9} and has undergone extensive cryptanalysis since its NIST
submission, with the full-round variant remaining unbroken as of 2024.

\textbf{ASCON software implementations.}
Kannwischer et al.~\cite{b10} benchmark ASCON and post-quantum primitives in
the pqm4 framework on ARM Cortex-M4, reporting 7.9 cycles/byte for the
optimized C implementation at 168\,MHz. The official ASCON C reference
team~\cite{b12} provides an optimized C/assembly implementation whose
design choices informed our Rust port. Weatherley~\cite{b11} provides an
Arduino-compatible C implementation used in industrial pilots but without
systematic performance characterization. None of these works provide a
memory-safe Rust implementation with a protocol-level design.

\textbf{Side-channel considerations.}
Moradi and Poschmann~\cite{b13} survey DPA countermeasures for lightweight
ciphers, and Kocher et al.~\cite{b15} established the DPA methodology. Xue
et al.~\cite{b14} demonstrate side-channel attacks against PRINCE. Our
implementation provides structural side-channel resistance (branchless S-box,
no secret-dependent memory access) but does not claim empirical TVLA results,
which require physical power-measurement hardware outside our scope.

\textbf{Memory safety in cryptography.}
The case for memory-safe languages in cryptographic implementations is
well-established~\cite{b16}. The RustCrypto project~\cite{b17} maintains
pure-Rust cryptographic primitives, including an \texttt{ascon-aead} crate,
but without a protocol-level design or systematic performance characterization
as a package. Our work differs in providing a complete, independently
implemented and verified AEAD + packet protocol stack.

\section{Background}
\label{sec:background}
% ══════════════════════════════════════════════════════════════════════════════

\subsection{ASCON-128 Authenticated Encryption}

ASCON~\cite{b9} is a family of algorithms based on a 320-bit permutation
denoted $p^{12}$ (12-round initialization/finalization) and $p^{6}$
(6-round data processing). The state is five 64-bit words
$[x_0, x_1, x_2, x_3, x_4]$. Each round applies:
\begin{enumerate}
  \item \textbf{AddConstants (AC):} $x_2 \leftarrow x_2 \oplus c_r$ for
        round constant $c_r$.
  \item \textbf{SubBytes ($\chi$):} A bitsliced 5-bit S-box applied via
        XOR/AND/NOT across all five words simultaneously, with zero table lookups.
  \item \textbf{Linear Diffusion ($\Sigma$):} Word-specific double rotations, e.g.,
        $x_0 \leftarrow x_0 \oplus \mathrm{ROR}(x_0, 19) \oplus \mathrm{ROR}(x_0, 28)$.
\end{enumerate}

For ASCON-128 AEAD with a 16-byte key $K$, 16-byte nonce $N$, associated data
$A$, and plaintext $P$:
\begin{enumerate}
  \item \textit{Init:} $S \leftarrow p^{12}(\mathit{IV} \| K \| N)$;
        $S[3..4] \leftarrow S[3..4] \oplus K$
  \item \textit{AD:} absorb $A$ in 8-byte blocks with $p^{6}$ between each
  \item \textit{Encrypt:} absorb $P$ in 8-byte blocks with $p^{6}$, extract $C$
        by XOR with rate
  \item \textit{Final:} $T \leftarrow \ell(p^{12}(S \oplus (0^* \| K))) \oplus K$
\end{enumerate}

The IV for ASCON-128 encodes algorithm parameters as $\mathtt{0x80400c0600000000}$.
The bitsliced S-box eliminates all data-dependent memory accesses, removing
the cache-timing vulnerability inherent in table-driven ciphers such as AES.

\subsection{Rust for Embedded Systems}

Rust enforces memory safety through a borrow checker at compile time,
eliminating null-pointer dereferences, buffer overflows, use-after-free, and
data races without a garbage collector. The \texttt{no\_std} attribute removes
the dependency on an operating system's standard library, enabling bare-metal
compilation for \texttt{thumbv7em-none-eabihf} (Cortex-M4) and similar targets.
The \texttt{heapless} crate provides stack-allocated data structures; the
\texttt{subtle} crate wraps conditional operations in constant-time selection
primitives; and the \texttt{zeroize} crate guarantees secure erasure of
sensitive data on drop.

\section{System Design}
\label{sec:design}

\subsection{Threat Model}

We consider a network adversary with full access to the wireless channel
(eavesdrop, replay, inject, modify) and passive physical access to sensor
nodes (power observation). The adversary does not know the 128-bit pre-shared
key stored in secure non-volatile memory. Security goals are:
(1)~\emph{Confidentiality}---IND-CCA2 under ASCON-128;
(2)~\emph{Authenticity}---forgery probability bounded by $2^{-128}$;
(3)~\emph{Replay resistance}---monotonic sequence counter checked at receiver;
(4)~\emph{Memory safety}---zero undefined behavior by construction.

The nonce is constructed as: 64-bit sequence counter (big-endian) $\|$
first 32 bits of ASCON-HASH(\texttt{device\_id}) $\|$ $0^{32}$. This ensures
nonce uniqueness across power cycles (counter) while binding the nonce to the
device identity. The associated data block (header $\|$ sequence counter) is
authenticated but not encrypted, protecting metadata integrity without
increasing ciphertext size.

\subsection{Software Architecture}

RustGuard is organized as a Cargo workspace with two independently usable crates:

\noindent\textbf{rustguard-core} contains the ASCON-128 permutation, AEAD
encrypt/decrypt, and ASCON-HASH. Its only dependencies are \texttt{subtle}
(constant-time selection) and \texttt{zeroize} (secure erasure). The crate
carries \texttt{\#![no\_std]} and \texttt{\#![forbid(unsafe\_code)]}, making
it suitable for direct cross-compilation to any embedded target.

\noindent\textbf{rustguard-pap} implements the \texttt{PacketBuilder} API and
the replay-resistant unwrapping logic. It depends only on
\texttt{rustguard-core} and \texttt{heapless}, adding no heap allocation.

\section{Implementation}
\label{sec:impl}
% ══════════════════════════════════════════════════════════════════════════════

\subsection{ASCON-128 Permutation Core}

The 320-bit state is represented as a \texttt{State} struct of five
\texttt{u64} fields, derived with \texttt{zeroize::Zeroize} and
\texttt{\#[zeroize(drop)]} for automatic key erasure. The S-box is
implemented as a branchless bitsliced sequence:

\begin{lstlisting}[language=C,caption={Bitsliced ASCON S-box (\texttt{rustguard-core/src/lib.rs}).}]
fn ascon_sbox(s: &mut State) {
    s.x0 ^= s.x4;  s.x4 ^= s.x3;  s.x2 ^= s.x1;
    let (t0,t1,t2,t3,t4) = (s.x0,s.x1,s.x2,s.x3,s.x4);
    s.x0=t0^(!t1&t2); s.x1=t1^(!t2&t3);
    s.x2=t2^(!t3&t4); s.x3=t3^(!t4&t0);
    s.x4=t4^(!t0&t1);
    s.x1^=s.x0; s.x0^=s.x4; s.x3^=s.x2; s.x2=!s.x2;
}
\end{lstlisting}

This implementation contains \emph{zero data-dependent branches} and
\emph{zero table lookups} on secret data, establishing structural resistance
to cache-timing attacks. All \texttt{if}/\texttt{else} constructs in the AEAD
logic operate only on public data (lengths, padding positions).

\subsection{Constant-Time Properties}

We verify constant-time behavior through source-level analysis:
the S-box and diffusion layer operate exclusively via XOR, AND, NOT, and
rotate instructions on the five state words. No conditional branch depends on
key or plaintext bits; no memory address is computed from secret data.
The \texttt{subtle::ConstantTimeEq::ct\_eq} method is used for the
authentication tag comparison (Listing~2), replacing a
potentially branch-predictable \texttt{==}:

\begin{lstlisting}[language=C,caption={Constant-time tag verification.}]
let ok = bool::from(tag.ct_eq(&expected));
if !ok { plaintext.zeroize(); }  // erase on failure
ok
\end{lstlisting}

\subsection{Packet Framing}

\texttt{PacketBuilder::build\_packet} assembles a packet in a
\texttt{heapless::Vec<u8, 552>} (maximum packet, stack-allocated) with no
heap allocation at any point. Nonce generation uses the sequence counter as
the high 64 bits, preventing nonce reuse as long as the counter is persisted
to non-volatile memory between power cycles~---~a standard recommendation for
symmetric AEAD in constrained environments~\cite{b9}.

\section{Verification}
\label{sec:verification}
% ══════════════════════════════════════════════════════════════════════════════

\subsection{Test Suite Structure}

The test suite comprises 24 automated tests organized in two layers.
The 15 unit tests in \texttt{rustguard-core} cover:
\begin{itemize}
  \item 5 AEAD round-trip tests (empty, 1-byte, 1-block, 2-block, partial-block)
  \item 4 authentication failure tests (ciphertext tamper, tag tamper, AD
        tamper, wrong key) --- each verifies that plaintext is zeroized on failure
  \item 3 security property tests (nonce uniqueness, determinism, and
        ciphertext/tag divergence under distinct nonces)
  \item 3 ASCON-HASH tests (determinism, collision resistance on adjacent
        inputs, multi-block absorption)
\end{itemize}

The 9 integration tests in \texttt{rustguard-pap} cover:
\begin{itemize}
  \item Round-trip correctness for all 7 payload sizes (8--512 bytes)
  \item Replay detection at boundary conditions
  \item Tamper detection on ciphertext, header, and tag fields
  \item Wrong-key authentication failure
  \item Minimum-size boundary check
\end{itemize}

All 24 tests pass (\texttt{cargo test --release}). Fig.~\ref{fig:tests}
summarizes results by category.

\begin{figure}[ht]
  \centering
  \includegraphics[width=\columnwidth]{fig6_tests.png}
  \caption{Test suite results: 24/24 tests pass across correctness, security,
           PAP protocol, and edge-case categories.}

\subsection{Memory Safety Guarantees}

The \texttt{\#![forbid(unsafe\_code)]} attribute causes compilation to fail
if any \texttt{unsafe} block is introduced, providing a compile-time
guarantee that the library is free of undefined behavior classes including
buffer overruns, dangling pointers, and uninitialized reads. This attribute is
enforced on both \texttt{rustguard-core} and \texttt{rustguard-pap}. Combined
with the \texttt{zeroize} implementation, key material cannot persist in memory
after the \texttt{State} struct goes out of scope.

\section{Performance Characterization}
\label{sec:perf}
% ══════════════════════════════════════════════════════════════════════════════

\subsection{Measurement Methodology}

All benchmarks were executed on an x86-64 host (Intel Core, standard laptop)
using Rust 1.75.0, \texttt{opt-level = 3}, \texttt{lto = true},
\texttt{codegen-units = 1}. Timing used \texttt{std::time::Instant} with
N\,=\,10{,}000 iterations per configuration, preceded by N\,=\,200 warmup
iterations. We report mean and one standard deviation. These are \emph{host
benchmarks}~---~not embedded measurements~---~and should be interpreted as a
software performance baseline, not as Cortex-M cycle counts. The associated
data for all AEAD measurements is a fixed 8-byte block (matching the PAP
header $\|$ sequence counter structure).

\subsection{AEAD Latency and Throughput}

Fig.~\ref{fig:latency} shows encrypt, decrypt, and full PAP packet latency
versus payload size. For a 32-byte payload (representative of IoT sensor data
with mean 34.7 bytes in the N-BaIoT dataset~\cite{b18}), encryption takes
$349.8 \pm 191.5$\,ns. At 512 bytes, latency reaches
$2{,}100.4 \pm 553.2$\,ns. The PAP overhead (nonce generation, header
assembly, and protocol framing) adds approximately 350\,ns above raw AEAD
latency for small payloads.

\begin{figure}[ht]
  \centering
  \includegraphics[width=\columnwidth]{fig1_latency.png}

Fig.~\ref{fig:throughput} shows throughput versus payload size. Raw ASCON-128

\subsection{Permutation Latency}

Fig.~\ref{fig:permutation} shows measured latency for the two permutation
variants. The $p^{12}$ (12-round) permutation used for initialization and
finalization takes $86.1 \pm 164.2$\,ns; $p^{6}$ (6-round data processing)
takes $58.5 \pm 26.1$\,ns. The ratio $86.1 / 58.5 = 1.47$ closely matches
the theoretical prediction of $12/6 = 2.0$, with the difference attributable
to instruction-level parallelism and CPU out-of-order execution that
pipeline successive rounds more efficiently at 12 rounds than 6.

\section{Discussion and Future Work}
\label{sec:discussion}
% ══════════════════════════════════════════════════════════════════════════════

\textbf{Hardware port.}
The most direct extension of this work is cross-compiling and benchmarking on
ARM Cortex-M hardware using the \texttt{thumbv7em-none-eabihf} target.
The \texttt{rustguard-hal} crate skeleton provides STM32L476 integration via
\texttt{stm32l4xx-hal}. Actual DWT cycle counting would yield the cycles/byte
figures comparable to pqm4~\cite{b10} and enable energy measurement via a
Nordic PPK2 current probe.

\textbf{TVLA side-channel evaluation.}
While the branchless S-box provides structural protection, empirical TVLA
validation~\cite{b19} using ChipWhisperer-Lite~---~a \$250 platform~---~would
confirm first-order DPA resistance. This is the most meaningful near-term
extension for a security audience.

\textbf{Second-order masking.}
Applying Boolean masking to the ASCON S-box would provide second-order DPA
resistance at approximately $2\times$ code size and $1.4\times$ execution time,
based on published masked ASCON implementations.

\textbf{ASCON-80pq and post-quantum hybrids.}
The ASCON-80pq variant uses a 160-bit key providing resistance against limited
quantum adversaries while reusing the same permutation. Integration with
CRYSTALS-Kyber~\cite{b20} for post-quantum key encapsulation is a natural
extension once hardware performance is characterized.

\textbf{Key management and provisioning.}
The current design uses pre-shared keys provisioned at manufacture. A full
deployment would require a key management infrastructure; lightweight
certificate-less authenticated key agreement protocols are an active research
area for IoT.

% ══════════════════════════════════════════════════════════════════════════════
\section{Conclusion}
\label{sec:conclusion}
% ══════════════════════════════════════════════════════════════════════════════

This paper presented RustGuard: a verified, memory-safe, \texttt{no\_std}
Rust implementation of ASCON-128 authenticated encryption and the RustGuard-PAP
packet authentication protocol. All 24 automated tests pass, covering AEAD
correctness, authentication failure, replay detection, and security properties.
The library is \texttt{\#![forbid(unsafe\_code)]} throughout, providing a
compile-time memory-safety guarantee absent from all prior ASCON
implementations. On x86-64, ASCON-128 encryption of a 32-byte payload
completes in $349.8 \pm 191.5$\,ns, with the 11{,}605-byte text segment
establishing a baseline for subsequent embedded porting. The complete,
buildable, and test-verified source code is available at
\url{https://github.com/ta08451/rustguard}.
